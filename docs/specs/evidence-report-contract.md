# Technical Evidence Report Contract

## Purpose

Defines the product/architecture contract for technical evidence reports. This is not a JSON schema or implementation code.

## Required Field Groups

| Field group | Required? | Purpose |
| --- | --- | --- |
| assessment_id | Yes | Map report to assessment |
| source_type | Yes | MVP: `GITHUB_REPOSITORY_SCAN`. Deferred/Future: `LOCAL_CI_REPORT`, `MANUAL_TECHNICAL_JSON`, `CLI_REPORT` |
| system_identifier | Yes | System/repo/API being assessed |
| provenance | Yes | Actor, provider, repo, branch, commit, environment |
| scanner_version | Required for scanner-generated reports | Reproducibility |
| ruleset_version | Required for scanner-generated reports | Detection rule trace |
| timestamp | Yes | Freshness |
| scope | Yes | Files/languages/tools/coverage |
| privacy_flags | Yes | No raw source/full AST/secrets handling violation |
| technical_profile | Yes | Normalized technical signals |
| findings | Yes, may be empty | Evidence items |
| confidence_per_signal | Yes | Signal confidence |
| report_hash | Yes | Integrity |
| scanner_contract_version | Yes for scanner-generated reports | Static scanner contract trace |
| graph_summary | Required when graph analysis ran | Counts/coverage for graph nodes, edges and evidence paths; no raw source |
| coverage_limitations | Yes, may be empty | Explicit scan coverage limitations and unsupported dynamic flows |

MVP scanner metadata fields:

| Field | Required? | Purpose |
| --- | --- | --- |
| scan_job_id | Yes | Trace report to repository scan job |
| repository_id | Yes | Identify connected GitHub repository |
| branch | Yes | Scan scope |
| commit_sha | Yes | Reproducible scan target |
| scan_started_at | Yes | Job audit |
| scan_completed_at | Required on completed scan | Job audit and freshness |
| scan_status | Yes | queued/running/completed/failed |

Optional fields:

- `report_version`
- `supporting_document_refs`
- `api_probe_refs`
- `attestation_refs`

## Static Analysis Scanner Boundary

For MVP, `TechnicalEvidenceReport` is generated by static analysis only. The scanner must not execute customer code, run install/build/test commands, execute Dockerfiles or CI workflows, probe APIs, persist raw source long-term, or send raw source/full prompts/secrets to LLM.

The authoritative scanner contract is:

```text
docs/specs/static-analysis-scanner-contract.md
```

## Graph Evidence Payload

When scanner graph analysis runs, report payload may include graph metadata and graph path refs:

| Field | Purpose |
| --- | --- |
| `graph_node_count` | Number of normalized graph nodes |
| `graph_edge_count` | Number of normalized graph edges |
| `graph_path_refs[]` | Evidence paths from route/service/model/output/action |
| `analysis_levels_used[]` | L0-L4 analysis levels used |
| `coverage_limitations[]` | `SCAN_COVERAGE_LIMITATION` / `UNSUPPORTED_DYNAMIC_FLOW` findings |

Graph payload must persist metadata, refs and hashes only. It must not persist raw source body or complete AST body.

## Technical Profile Minimum Dimensions

| Dimension | Required status |
| --- | --- |
| ai_usage_signals | DETECTED, NOT_DETECTED, UNKNOWN |
| data_type_signals | DETECTED, NOT_DETECTED, UNKNOWN |
| decision_flow_signals | DETECTED, NOT_DETECTED, UNKNOWN |
| human_oversight_signals | DETECTED, NOT_DETECTED, UNKNOWN |

## AI Usage Flow Analysis

AI Usage Flow Analysis identifies the purpose, role, impact, and harm potential of AI usage inside the scanned repository flow. It connects technical AI signals to business process meaning so that legal rules can be matched correctly.

Trong LCSP, `TechnicalProfile` trả lời AI được dùng như thế nào về mặt kỹ thuật, còn `AIUsageFlow` trả lời AI được dùng để làm gì trong luồng nghiệp vụ, output AI ảnh hưởng quyết định nào, ai bị ảnh hưởng và có khả năng gây hại hay không.

## AI Usage Finding Types

Scanner/normalizer should produce or preserve the following finding types when detected. These findings are evidence inputs for AI Usage Flow Analysis; they are not legal conclusions.

| Finding Type | Ý nghĩa |
| --- | --- |
| `AI_INPUT_SIGNAL` | Dữ liệu nào được đưa vào AI |
| `AI_OUTPUT_SIGNAL` | AI tạo output gì |
| `AI_DECISION_FLOW_SIGNAL` | Output AI được dùng trong decision/workflow |
| `AUTOMATED_DECISION_SIGNAL` | AI output dẫn tới quyết định tự động |
| `HUMAN_REVIEW_SIGNAL` | Có bước con người review output AI |
| `RANKING_SIGNAL` | AI output tham gia ranking/sorting |
| `RECOMMENDATION_SIGNAL` | AI output tạo recommendation |
| `STATUS_UPDATE_SIGNAL` | AI output dẫn tới status update |
| `USER_IMPACT_SIGNAL` | Output AI ảnh hưởng tới người dùng/customer/employee/student/patient |
| `SENSITIVE_DATA_SIGNAL` | Có dữ liệu cá nhân/nhạy cảm/sinh trắc/y tế/tài chính |
| `DOMAIN_CONTEXT_SIGNAL` | Lĩnh vực nghiệp vụ: HR, finance, healthcare, education, legal, public service |
| `SYSTEM_PROMPT_PURPOSE_SIGNAL` | System prompt thể hiện mục đích AI |
| `HARM_POTENTIAL_SIGNAL` | Có dấu hiệu AI có thể gây ảnh hưởng tiêu cực |
| `SCAN_COVERAGE_LIMITATION` | Scanner không bao phủ được một phần source/flow |
| `UNSUPPORTED_DYNAMIC_FLOW` | Flow vượt qua dynamic/unsupported boundary |

## AIUsageFlow Output Contract

`AIUsageFlow` is derived after `TechnicalProfile` and before Reconciliation.

| Field | Purpose |
| --- | --- |
| `ai_usage_flow_id` | Stable identifier for the analysis result |
| `assessment_id` | Assessment mapping |
| `source` | MVP: `REPOSITORY_SCAN` |
| `business_process` | loan_approval, recruitment, customer_support, content_generation, education, healthcare, unknown |
| `ai_purpose` | summarization, recommendation, scoring, classification, decision_support, automated_decision, chatbot, unknown |
| `ai_input_types` | user_input, personal_data, financial_data, sensitive_data, unknown |
| `ai_output_types` | score, recommendation, decision_label, generated_text, summary, unknown |
| `downstream_action` | approve_reject, rank, recommend, notify, generate_content, unknown |
| `affected_subjects` | customer, employee, student, patient, citizen, internal_user, unknown |
| `human_review` | present, absent, unclear |
| `automation_level` | assistive, decision_support, semi_automated, fully_automated, unclear |
| `potential_harm_categories` | denial_of_service, financial_loss, discrimination, privacy_risk, misinformation, other |
| `evidence_refs` | Finding ids supporting each usage-flow conclusion |
| `confidence` | 0.0-1.0 confidence in the usage-flow analysis |
| `claim_evidence` | Claim-level evidence refs for business_process, ai_purpose, input/output/action/review/harm claims |

If usage flow cannot be inferred confidently, fields must be marked `unknown` or `unclear`. LCSP must request confirmation or create reconciliation conflict rather than overclaiming risk.

## Schema Completeness Gate

Reject report if:

- Missing assessment mapping.
- Missing source type/system identifier.
- Missing provenance.
- Missing required scanner/report version.
- Missing timestamp/scope/privacy flags.
- Missing technical profile or findings array.
- Missing report hash.
- Privacy flags indicate raw source sent to LLM or long-term raw source storage.

## Quality Threshold Gate

Accepting a report is not the same as unlocking classification.

| Condition | Result |
| --- | --- |
| Schema fail | TECHNICAL_EVIDENCE_REJECTED |
| Schema pass but quality/context insufficient | TECHNICAL_EVIDENCE_INSUFFICIENT |
| Schema pass and quality threshold pass | TECHNICAL_EVIDENCE_READY |

Quality threshold is context-aware and depends on WizardProfile. A1 validation may change required Wizard fields and therefore threshold logic.

## Accepted Evidence Condition

Evidence can participate in reconciliation when:

- Schema completeness gate passes.
- Privacy flags pass.
- Provenance is present.
- Technical profile includes required dimensions.
- Evidence quality is sufficient or supplemented by valid controlled evidence.

## Insufficient Evidence Condition

Evidence remains insufficient when:

- Critical fields are UNKNOWN/low-confidence for a high-impact context.
- Scope coverage is too low.
- Provenance is stale or incomplete but not schema-invalid.
- Scanner evidence is weak and no valid supplemental evidence exists.

## Guardrails

- Scanner evidence creates technical evidence; it does not create legal conclusions.
- Manager may initiate GitHub repository connection and Repository Scan for own assessment in MVP.
- Developer may initiate repository scan only when delegated by Manager; Developer assignment is not required for MVP evidence collection.
- Scanner/model/provider/framework detection alone must not create risk classification.
- AI Usage Flow Analysis must cite evidence refs for purpose, input, output, downstream action and affected subject claims.
- AIUsageFlow claim without evidence refs cannot be used for legal rule matching.
- Raw source code must not be sent to LLM.
- Raw source code must not be stored long-term.
- Findings must not include long source snippets.
