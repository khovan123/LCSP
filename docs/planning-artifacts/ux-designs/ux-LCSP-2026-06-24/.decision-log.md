## Decisions

1. Use a quiet operational web-app posture for LCSP.
   - Rationale: LCSP is an evidence-based compliance support product, not a marketing site, chatbot, or consumer assistant.

2. Treat Manager as the primary protagonist and Developer as optional scoped collaborator.
   - Rationale: Phase 5.2L requires Manager-completable golden path without Developer participation.

3. Make evidence gates and blocked states first-class UX objects.
   - Rationale: The product must prevent Wizard-only risk levels, scanner bypass, unsupported legal certainty, and report overclaim.

4. Keep legal corpus administration outside customer product UX.
   - Rationale: Active docs define corpus ingestion/review/approval as internal API/CLI-only for MVP.

5. Carry ChromaDB/vectorless legal RAG into UX only through citation/provenance behavior.
   - Rationale: The backend retrieval path does not change high-level flow, but the UI must show document/article/clause/point citations, corpus version/effective date, primary versus referenced context, expired/superseded warnings, and no citation outside the retrieved allowlist.

6. Exclude structured attestation, manual technical evidence JSON upload, Local/CI scanner report upload, and delegated free-form clarification from active screens.
   - Rationale: Phase 5.2L marks those as superseded, removed, or deferred.

## Assumptions

- [ASSUMPTION] The web frontend will use a modern component library with table, dialog, sheet, tabs, toast, alert, badge, stepper, and command/search primitives; no specific library is confirmed in active docs.
- [ASSUMPTION] Vietnamese is the primary product language for this UX run; English identifiers may remain visible only as technical/audit references.
- [ASSUMPTION] Desktop/laptop web is the primary surface, with responsive support for tablet and phone review workflows.
- [ASSUMPTION] No final brand identity exists yet, so DESIGN.md defines a conservative product design system rather than a polished brand campaign.
