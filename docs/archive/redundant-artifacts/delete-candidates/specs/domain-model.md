# LCSP Domain Model

## Purpose

This document defines LCSP domain objects at conceptual level before implementation. It is not a database schema and does not create implementation tasks.

Conditional dependencies:

- A1 may change WizardProfile fields.
- A2 may change LegalRule and LegalCitation shape.
- A3 may change HumanAttestation and ConflictResolution rules.

## Domain Objects

| Object | Purpose | Key fields | Owner module | Lifecycle | Important constraints |
| --- | --- | --- | --- | --- | --- |
| User | LCSP user identity | user_id, organization_id, email, auth_provider, mfa_enabled, failed_mfa_attempts, locked_until | Authentication | Registered/invited; authenticates into workspaces | MFA-enabled users must pass Authenticator App OTP before session; auth events audited |
| OAuthIdentity | Linked OAuth/OIDC identity | oauth_provider, provider_subject, provider_email, email_verified_at, linked_at, last_login_at, token_metadata_reference | Authentication | Created/linked after safe OAuth/OIDC callback validation | Does not store raw provider access token; does not grant repository access |
| UserMfaMethod | Authenticator App MFA enrollment | id, user_id, method_type, secret_reference, enabled, enabled_at, last_verified_at | Authentication | Setup pending -> enabled -> disabled/reset | MFA secret must not be stored/logged plaintext; reset requires recovery policy |
| Assessment | Root compliance assessment record | id, organization_id, manager_id, status, system_name, created_at | Assessment | Created by Manager; progresses through evidence/reconciliation/classification/report states | Manager owns assessment; no risk level without evidence |
| Manager | Superset MVP role and assessment owner | user_id, role, organization_id | Role/Permission | Invited or created in org; owns assessment | Can complete MVP assessment end-to-end, including repository scan, conflict resolution, VerifiedProfile approval and report |
| Developer | Optional technical collaborator assigned by Manager | user_id, role, organization_id, policies | Role/Permission | Invited/delegated to assessment; works on scoped technical tasks | Only sees assigned task/policy scope; not required for MVP success path |
| PermissionGrant | Post-MVP delegated permission | granted_by_manager_id, grantee_user_id, scope_type, scope_id, permission_code, status, expires_at, revoked_at | Role/Permission | Created/revoked by Manager | Delegation never removes Manager permissions; all grants/revocations audited |
| WizardProfile | Business/legal self-declared profile | purpose, sector, data_type, user_group, user_impact, decision_role, human_oversight, external_llm_usage | Wizard | Created after Wizard submission; reconciled with TechnicalProfile | Wizard-only cannot produce risk level |
| TechnicalEvidence | Technical evidence package | report_id, source_type, provenance, report_hash, gate_status | Evidence | MVP: generated by GitHub Repository Scan; Future: other source paths may be evaluated | Must pass schema/privacy/provenance gates |
| StaticAnalysisScan | Static scanner run | scan_id, repository_id, commit_sha, scanner_version, ruleset_version, analysis_levels_used, idempotency_key, coverage_summary | Scanner/Evidence | Created for each Repository Scan job | Static-analysis only; no source execution; no raw source persistence |
| ScannerGraph | Scan-local evidence graph metadata | graph_node_refs, graph_edge_refs, graph_path_refs, confidence, coverage_limitations | Scanner/Evidence | Built in memory during scan, persisted as normalized metadata/refs | NetworkX graph is scan-local; persisted graph excludes source bodies and full AST bodies |
| TechnicalProfile | Normalized technical summary | ai_usage_signals, data_type_signals, decision_flow_signals, human_oversight_signals | Evidence | Derived from evidence reports | Does not make legal conclusions |
| AIUsageFlow | Business-flow interpretation of AI usage | business_process, ai_purpose, ai_input_types, ai_output_types, downstream_action, affected_subjects, human_review, automation_level, potential_harm_categories, evidence_refs, claim_evidence, confidence | AI Usage Flow / Evidence | Created after TechnicalProfile and before Reconciliation | Bridges technical evidence to business/legal meaning; each material claim needs evidence refs; must not classify risk or invent legal conclusions |
| EvidenceFinding | Specific technical signal | id, evidence_type, source_tool, confidence, refs | Evidence | Generated by scanner/normalizer or accepted report | Must avoid long raw source snippets |
| EvidenceGateResult | Evidence validation outcome | schema_status, quality_status, status, reasons | Evidence Gate | Created after report validation | Low quality alone cannot unlock classification |
| Conflict | Mismatch between WizardProfile and TechnicalProfile/AIUsageFlow | field, wizard_value, technical_value, usage_flow_value, score, severity, resolver | Reconciliation | Created during reconciliation | Any unresolved conflict blocks classification/final report in MVP; severity is informational only |
| ConflictResolution | Human resolution of conflict | conflict_id, actor_id, role, decision, reason, timestamp | Reconciliation | Created by Manager resolution; may reference optional Developer clarification | Manager is final resolver for MVP conflicts |
| HumanAttestation | Structured human evidence supplement | source_type, role, claim, reason, scope, timestamp, audit_ref | Evidence/Reconciliation | Submitted by authorized role | Cannot replace machine-generated metadata |
| VerifiedProfile | Canonical profile for classification | version, wizard_refs, evidence_refs, ai_usage_flow_ref, resolutions, approved_by | Reconciliation | Created after evidence gates, AI Usage Flow Analysis and conflict resolution | Required before Risk Classification |
| RiskClassificationResult | Risk output with legal trace | risk_level, rule_refs, legal_refs, confidence, status | Classification | Created after VerifiedProfile and legal retrieval | Must cite legal rules; block/degrade if citation missing |
| GapAnalysisResult | Compliance obligations/gaps | assessment_id, risk_ref, gaps, obligations, citations | Gap Analysis | Created after classification | Not generated from Wizard-only readiness |
| ComplianceDocument | Generated compliance/report artifact | document_id, type, version, storage_ref, generated_at | Document | Generated after valid classification/gap analysis | No final report with any unresolved conflict |
| AuditEvent | Append-oriented trace event | event_id, actor, action, target, timestamp, metadata | Audit | Created throughout lifecycle | Must not store raw source |
| LegalRule | Structured compliance rule | rule_id, condition, outcome, source_ref, version | Legal Corpus | Curated/versioned before classification | LLM cannot invent missing rule |
| LegalCitation | Legal source reference | citation_id, legal_source, section, version, effective_date | Legal Corpus | Linked to LegalRule and output | Required for final classification on critical rules |

## Cross-object Rules

- `UserMfaMethod` uses Authenticator App/TOTP flow; SMS OTP is not in MVP scope.
- `User` session creation requires valid MFA verification when `mfa_enabled=true`.
- `OAuthIdentity` authenticates LCSP user identity only. It does not connect GitHub repository, grant repository scan permission or replace LCSP RBAC.
- `PermissionGrant` is post-MVP delegation; Manager keeps all active permissions and may revoke or perform delegated actions at any time.
- `RiskClassificationResult` requires `VerifiedProfile`.
- `VerifiedProfile` requires valid evidence, AIUsageFlow with evidence refs or explicit uncertainty resolution, and no unresolved conflict.
- `AIUsageFlow` must not classify risk by itself. It explains business process, AI purpose, input/output, downstream action, affected subjects, human review, automation level and harm potential.
- `StaticAnalysisScan` must not execute repository code, run install/build/test commands, probe APIs, persist raw source long-term or send raw source/full prompts/secrets to LLM.
- `ScannerGraph` supports traceable AIUsageFlow claims but does not persist raw source body or complete AST body.
- Risk classification must not rely only on provider/model/framework presence.
- `HumanAttestation` can supplement evidence but cannot replace report hash, scanner version, ruleset version, scan timestamp, repo/commit metadata, legal corpus version, evidence integrity or machine-generated privacy flags.
- `AuditEvent` must trace all gate transitions, conflict resolutions, attestations, classification and document generation.
